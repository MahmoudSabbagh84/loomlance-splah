# Security Hardening Checklist (owner action — AWS console)

From the 2026-06-30 technical audit. These are **infra changes on CloudFront**, so they can't be
done from the repo — you (the account owner) apply them once in the AWS console. Everything here is
passive-hardening; the audit found **no leaked secrets**. Effort: ~15–20 minutes total.

Two CloudFront distributions are in scope:
- **loomlance.com** (marketing site) — lower stakes.
- **app.loomlance.com** (the dashboard SPA) — **higher priority**: this is where authenticated user
  and payment data lives. Apply everything here first.

---

## 1. Response Headers Policy (adds the six missing security headers)

Both distributions currently return none of these. Fix with one reusable policy.

**Console path:** CloudFront → Policies → Response headers → **Create response headers policy**.
Then attach it to each distribution's default behavior (Behaviors → Edit → Response headers policy).

Set these values:

| Header | Value |
|---|---|
| `Strict-Transport-Security` | `max-age=63072000; includeSubDomains; preload` |
| `X-Content-Type-Options` | `nosniff` |
| `X-Frame-Options` | `DENY` |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` |
| `Content-Security-Policy` | see below — set per-site, test in Report-Only first |

### CSP — start in Report-Only, then enforce
CSP breaks the page if a real script source is missing, so **first** set it as
`Content-Security-Policy-Report-Only` (the browser logs violations but doesn't block), watch the
console for a day, add anything legitimate it flags, **then** rename the header to
`Content-Security-Policy` to enforce.

**loomlance.com** (marketing — known sources: Google Tag Manager/Analytics, Google Fonts, unpkg for lucide):
```
default-src 'self';
script-src 'self' https://www.googletagmanager.com https://unpkg.com 'unsafe-inline';
style-src 'self' https://fonts.googleapis.com 'unsafe-inline';
font-src 'self' https://fonts.gstatic.com;
img-src 'self' data: https:;
connect-src 'self' https://www.google-analytics.com;
frame-ancestors 'none'; base-uri 'self';
```
(`'unsafe-inline'` is needed because the gtag snippet and JSON-LD are inline; tightening to a nonce
is a later refinement. Note: lucide is now pinned + SRI'd, which is the bigger supply-chain win.)

**app.loomlance.com** (dashboard SPA — add Supabase + Stripe, drop analytics if not used there):
```
default-src 'self';
script-src 'self' https://js.stripe.com;
style-src 'self' 'unsafe-inline';
img-src 'self' data: https:;
connect-src 'self' https://zbipqfsqxnvrzhpdjvvy.supabase.co https://api.stripe.com;
frame-src https://js.stripe.com https://hooks.stripe.com;
frame-ancestors 'none'; base-uri 'self';
```
Adjust `connect-src`/`frame-src` to the exact Supabase project URL and any other hosts the app calls
(check the browser Network tab against Report-Only violations before enforcing).

## 2. `/.well-known/security.txt`

- **loomlance.com:** already committed to this repo (`.well-known/security.txt`) and deploys with the
  site — verify it serves at https://loomlance.com/.well-known/security.txt after deploy.
- **app.loomlance.com:** add the same file to the dashboard's build output (or a CloudFront function),
  since researchers may look there too.

## 3. robots.txt / sitemap.xml (marketing site) — DONE in repo

- `robots.txt` — was a 404/HTML fallback; now a real file in this repo pointing at the sitemap.
- `sitemap.xml` — already correct (generated by the blog build).
- Nothing to do in the console unless the SPA fallback still shadows them (verify both return their
  real content-types post-deploy; if not, add a CloudFront behavior that serves `/robots.txt` and
  `/sitemap.xml` from S3 before the SPA index.html fallback).

## Verify after applying

```bash
curl -sI https://app.loomlance.com/ | grep -iE 'strict-transport|content-security|x-frame|x-content-type|referrer-policy|permissions-policy'
curl -sI https://loomlance.com/     | grep -iE 'strict-transport|content-security|x-frame|x-content-type|referrer-policy|permissions-policy'
curl -s  https://loomlance.com/.well-known/security.txt
```
All six headers should appear; security.txt should return its text.

> Beyond this passive pass: before the app handles real payment volume, commission an **active**
> security review of the authenticated dashboard (auth flows, RLS coverage, the Stripe webhook
> surface). That's out of scope for a headers-only hardening.
